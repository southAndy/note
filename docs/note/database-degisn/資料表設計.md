
#database 

資料表設計的好壞攸關 **資料讀取效率**、 **後續維護的便利性**，因此在開始著手前，要認真思考情境。


## 設計思考方式

1. 解析案子的業務流程，**將後端要支援的每個功能**，拆解成「需要哪些資料」與「資料之間的關係」。
2. 找出專案資料的 [[核心實體]]
3. 定義資料之間的關聯： [[鍵]]
4. [[資料正規化]]

## 轉為 Migration

因為手動撰寫 DDL 花時間且當表的架構更動時，不容易追蹤，所以後端框架有提供套件去讓你使用後端語言去進行資料庫操作，這行為就是所謂的 [[migration]]
